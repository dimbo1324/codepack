## Этапы реализации

### Этап 1 — Русификация интерфейса

**Ветка:** `feature/russian-ui`

**Задача:** перевести весь UI на русский язык.

**Что делаем:**
1. Проходим по всем файлам в `gui/pages/`, `gui/components/`, `gui/dialogs.py`, `gui/main_window.py`.
2. Переводим все строки: метки, кнопки, подсказки, заголовки, сообщения об ошибках.
3. Переводим строки логов, которые пользователь видит в `run_page.py`.
4. При необходимости уточняем ширину кнопок под русский текст (русские строки длиннее английских).

**Критерий готовности:** запустил приложение — ни одной английской строки в UI.

### Этап 2 — Авто-определение стека

**Ветка:** `feature/stack-detector`

**Задача:** приложение само понимает, какой проект открыт, и применяет правильные ignore-правила.

**Что делаем:**
1. Создаём `services/stack_detector.py` — анализирует корневую папку по маркерным файлам.
2. Создаём словарь ignore-шаблонов для каждого стека (`constants/stack_ignores.py`).
3. На странице «Проект» показываем определённый стек (метка + иконка).
4. Ignore-правила стека автоматически подгружаются на страницу «Безопасность».
5. Пользователь может изменить определённый стек вручную.

**Поддерживаемые стеки:** Node.js, Python, Go, Rust, Java/Maven, Java/Gradle, .NET/C#, Flutter, PHP, Ruby, iOS/Swift, Android.

**Критерий готовности:** открыл Node.js проект → `node_modules` исключён автоматически, без ручной настройки.

### Этап 3 — Интерактивное дерево файлов (страница Preview)

**Ветка:** `feature/preview-page`

**Задача:** показать пользователю, что войдёт в экспорт, до нажатия «Запустить».

**Что делаем:**
1. Добавляем страницу `gui/pages/preview_page.py` в навигацию между «Безопасность» и «Запуск».
2. Строим `QTreeWidget` на основе текущего `ExportPlan`.
3. Цветовая кодировка узлов: зелёный / красный / жёлтый / серый.
4. Всплывающая подсказка на узле — причина включения / исключения.
5. Двойной клик на файле — переключить include/exclude вручную (пишем в `ExportPlan.overrides`).
6. Кнопка «Сбросить ручные правки».
7. Статистика над деревом: N файлов включено, N исключено, ~N МБ.

**Критерий готовности:** дерево отображается за <2 секунды для проекта с 5000 файлами; ручные правки работают и сохраняются до нажатия «Запустить».

### Этап 4 — Счётчик токенов и Clipboard-экспорт

**Ветка:** `feature/token-counter-clipboard`

**Задача:** дать разработчику понять, «влезет ли» дамп в контекст ИИ; позволить копировать без создания архива.

**Что делаем:**

**Счётчик токенов:**
1. Создаём `services/token_counter.py` — оценка по символам (~4 символа = 1 токен для русского/английского кода, уточняемая эвристика).
2. Виджет-панель «Оценка токенов» на странице Preview и Result.
3. Строки для каждой модели: лимит, текущая оценка, статус (ОК / превышение).
4. Обновление в реальном времени при изменении набора файлов.

**Clipboard-экспорт:**
1. Кнопка «Скопировать дамп» на главном экране рядом с «Создать экспорт».
2. При нажатии — запускает `ExportWorker` в режиме `clipboard_only=True`.
3. Генерирует text dump без создания архива.
4. Помещает результат в `QApplication.clipboard()`.
5. Показывает уведомление: «Скопировано — ~42 000 токенов».

**Критерий готовности:** нажал «Скопировать дамп», вставил в Claude — всё работает.

### Этап 5 — Контекст разработчика и AI-пресеты

**Ветка:** `feature/developer-context-presets`

**Задача:** дать разработчику «голос» в экспорте; предложить готовые конфигурации под конкретные задачи.

**Что делаем:**

**Контекст разработчика:**
1. На странице «Проект» добавляем `QPlainTextEdit` — поле «Контекст / заметка».
2. Плейсхолдер: «Что вы сейчас делаете? Что нужно от ИИ? Что сломано?».
3. Текст сохраняется в `Config` для каждого проекта.
4. При генерации `ai_context_report.txt` текст вставляется первым блоком.

**AI-пресеты:**
1. В настройках — выпадающий список «AI-пресет» над профилями.
2. При выборе пресет перезаписывает профиль, режим безопасности, параметры отчётов.
3. Пресеты: «Claude Code», «ChatGPT», «Code Review», «Security Audit», «Онбординг».
4. Пользователь может сохранить текущие настройки как свой пресет.

**Критерий готовности:** выбрал «Claude Code» → настройки выставились автоматически, контекст из заметки появился в AI-файле.

### Этап 6 — Дифференциальный экспорт

**Ветка:** `feature/diff-export`

**Задача:** экспортировать только изменения, а не весь проект — для итеративной работы с ИИ.

**Что делаем:**
1. Создаём `services/diff_service.py`.
2. Три режима diff:
   - **С последнего экспорта** — сравниваем текущие хэши файлов с сохранёнными в `history.json`.
   - **С Git-ссылки** — `git diff --name-only <ref>` (HEAD~N, ветка, хэш).
   - **Только uncommitted** — `git status --porcelain`.
3. На странице «Настройки» — переключатель «Режим экспорта: Полный / Дифференциальный».
4. При выборе дифференциального — выбор ссылки и предпросмотр списка изменений.
5. В манифест добавляется поле `diff_base` с описанием точки сравнения.
6. Удалённые файлы фиксируются в `manifest.json` с пометкой `"status": "deleted"`.

**Критерий готовности:** выбрал «С последнего экспорта» → архив содержит только 3 изменённых файла, не весь проект.

### Этап 7 — Аналитика в UI

**Ветка:** `feature/analytics-ui`

**Задача:** показывать метрики проекта прямо в приложении, без открытия отчётов.

**Что делаем:**

2. Блок «Обзор проекта»: стек, языки, LOC, количество файлов, размер.

6. Блок «Риски»: найденные потенциальные секреты, подозрительные файлы.

**Критерий готовности:** открыл проект → на странице «Аналитика» видно распределение кода по языкам и список зависимостей.

### Этап 8 — Системный трей и Watch-режим

**Ветка:** `feature/tray-watch`

**Задача:** быстрый доступ к экспорту без открытия главного окна; автообновление при изменении файлов.

**Что делаем:**

**Системный трей:**
1. `QSystemTrayIcon` с иконкой приложения.
2. При закрытии окна — сворачиваем в трей, не выходим.
3. Контекстное меню трея: «Быстрый экспорт», «Открыть», «Выход».
4. «Быстрый экспорт» — запускает экспорт последнего проекта с последними настройками, уведомление по завершении.

**Watch-режим:**
1. Используем `QFileSystemWatcher` для мониторинга папки проекта.
2. При изменении файлов — обновляем внутренний план экспорта.
3. Опционально — автоматически обновляем clipboard-дамп (если пользователь включил).
4. Уведомление в трее: «Проект изменился: 3 файла».

**Критерий готовности:** свернул окно → иконка в трее; правой кнопкой → «Быстрый экспорт» работает.

### Этап 9 — История, сравнение снапшотов, темы

**Ветка:** `feature/history-comparison-themes`

**Задача:** улучшить работу с историей и добавить полноценную поддержку тем.

**Что делаем:**

**История:**
1. Улучшаем страницу истории: таблица с сортировкой и поиском.
2. Для каждой записи: дата, проект, пресет, файлы, токены, статус, кнопки действий.
3. Быстрый повторный экспорт с теми же настройками.

**Сравнение снапшотов:**
1. Кнопка «Сравнить» при выборе двух записей истории.
2. Диалог сравнения: добавленные / изменённые / удалённые файлы, изменение LOC.
3. Можно экспортировать diff между двумя снапшотами.

**Темы:**
1. Переключатель «Тёмная / Светлая / Системная» в настройках.
2. Два набора QSS-стилей (`app_dark.qss`, `app_light.qss`).
3. «Системная» — читает `QApplication.styleHints().colorScheme()`.
4. Тема сохраняется в `settings.json`.

**Критерий готовности:** выбрал светлую тему → всё приложение перекрасилось без перезапуска.

## Правила разработки

1. Каждый этап — отдельная ветка; merge в `main` только после прохождения тестов.
2. Новая функция не ломает существующий экспортный pipeline.
3. Добавление нового отчёта не требует изменений в UI.
4. UI вызывает только сервисы — не работает с файлами и архивами напрямую.
5. Любой вопрос «нужно ли это делать?» проверяется критерием: «Помогает ли это безопасно создать AI-ready снимок проекта или анализировать проект разработчику?»
