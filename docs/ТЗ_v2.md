## 1. Суть проекта

Project Exporter Desktop — настольная программа для Windows, которая создаёт структурированный, безопасный снимок любого программного проекта.

Снимок предназначен для:
- передачи ИИ-инструментам (Claude Code, ChatGPT, Codex, Gemini);
- передачи другому разработчику или команде;
- архивного хранения состояния проекта;
- анализа структуры, зависимостей и качества кода.

Главный принцип:

> Выбрал проект → увидел, что войдёт → нажал кнопку → получил понятный, безопасный, полезный архив.

Приложение устанавливается через `setup.exe` и используется как полноценная Windows-программа, а не скрипт из консоли.

---

## 2. Новый вектор развития

По сравнению с версией 1.0, проект меняет фокус: из инструмента для упаковки проекта в инструмент для разработчика, который используется ежедневно.

**Было:** «пакетировщик для ИИ» — выбрал, настроил, экспортировал.  
**Стало:** «рабочая среда разработчика» — открыл, увидел метрики, скопировал дифф, передал ИИ, продолжил работу.

Три ключевых сдвига:

1. **Умный импорт вместо ручной настройки.** Приложение само определяет стек и подставляет правильные ignore-правила. Разработчик видит результат, а не настраивает с нуля.

2. **Итеративная работа с ИИ.** Не каждый раз отправлять весь проект — только изменения. Дифференциальный экспорт и быстрый clipboard меняют ежедневный workflow.

3. **Аналитика для разработчика.** Метрики, граф зависимостей, история снапшотов — ценность не только при передаче, но и при самостоятельном анализе.

---

## 3. Целевая аудитория

1. Индивидуальный разработчик на любом стеке.
2. Разработчик, активно использующий ИИ-инструменты (Claude Code, ChatGPT, Codex).
3. Технический специалист, которому нужно быстро передать проект на ревью.
4. Создатель прототипов и MVP.

Пользователь не должен знать Python, устройство архивов, правила исключения секретов или особенности подготовки проекта для ИИ.

---

## 4. Что проект не делает

На текущем этапе не реализуется:
- облачная синхронизация и серверная часть;
- пользовательские аккаунты и подписки;
- автоматическая загрузка архивов в ИИ через API;
- редактор кода или IDE;
- полноценный Git-клиент;
- кроссплатформенность (macOS / Linux) — только Windows.

---

## 5. Основной пользовательский сценарий

1. Пользователь открывает приложение.
2. Выбирает папку проекта.
3. Приложение определяет стек, показывает интерактивное дерево файлов.
4. Пользователь видит: сколько файлов войдёт, сколько токенов займёт, есть ли секреты.
5. При необходимости — редактирует список файлов, добавляет заметку.
6. Нажимает «Создать экспорт» или «Скопировать дамп».
7. Получает архив с отчётами или текст в буфере обмена.
8. История сохраняется, следующий раз — быстрый повторный запуск.

## 7. Функциональные требования

### 7.1 Выбор проекта

- Выбор папки через диалог или перетаскивание (drag-and-drop).
- Проверка: путь существует, является директорией, не является системной папкой.
- Нормализация длинных путей Windows.
- История последних 10 проектов с быстрым доступом.

### 7.2 Авто-определение стека

Приложение анализирует корневую папку и определяет стек по наличию ключевых файлов:

| Стек | Маркеры |
|---|---|
| Node.js / npm | `package.json` |
| Python | `requirements.txt`, `pyproject.toml`, `setup.py` |
| Go | `go.mod` |
| Rust | `Cargo.toml` |
| Java / Maven | `pom.xml` |
| Java / Gradle | `build.gradle`, `settings.gradle` |
| .NET / C# | `*.csproj`, `*.sln` |
| Flutter / Dart | `pubspec.yaml` |
| PHP / Composer | `composer.json` |
| Ruby | `Gemfile` |
| iOS / Swift | `Package.swift`, `*.xcodeproj` |
| Android | `AndroidManifest.xml` |

Для каждого стека — готовый набор ignore-правил (директории и расширения). Пользователь видит обнаруженный стек и может скорректировать список.

### 7.3 Интерактивное дерево файлов (страница Preview)

До запуска экспорта пользователь видит дерево проекта:

- **Зелёный** — файл включён в экспорт.
- **Красный** — файл исключён (правило, секрет, бинарник).
- **Жёлтый** — файл с предупреждением (большой, подозрительный).
- **Серый** — игнорируется по умолчанию (пакеты, кэш).

Возможности:
- Кликнуть по файлу → включить / исключить вручную.
- Кликнуть по директории → включить / исключить всю ветку.
- Показ причины исключения во всплывающей подсказке.
- Кнопка «Сбросить к автоматическим правилам».

### 7.4 Счётчик токенов

На странице Preview и на странице результата отображается:

```
Оценка токенов текстового дампа:
  Claude 3.5 / Claude 4    ✅  ~42 000 / 200 000 токенов
  GPT-4 Turbo              ✅  ~42 000 / 128 000 токенов
  GPT-4o                   ✅  ~42 000 / 128 000 токенов
  GPT-3.5                  ⚠️  ~42 000 / 16 000 — превышает лимит
  Gemini 1.5 Pro           ✅  ~42 000 / 1 000 000 токенов
```

Алгоритм: оценка по количеству слов / символов (примерная, без API-запросов). Обновляется в реальном времени при изменении набора файлов.

### 7.5 Быстрый clipboard-экспорт

Кнопка «Скопировать дамп» на главном экране:

1. Генерирует текстовый дамп всех включённых файлов.
2. Копирует результат в буфер обмена Windows.
3. Показывает уведомление: «Скопировано: 42 000 токенов».

Не создаёт архив, не сохраняет файлы на диск. Предназначен для быстрой вставки в чат с ИИ.

### 7.6 Контекст разработчика

На странице выбора проекта или настроек — текстовое поле «Контекст»:

- Пользователь описывает, что он сейчас делает, что нужно от ИИ, что сломано.
- Текст попадает в начало AI-контекстного файла перед всем остальным.
- Сохраняется между сессиями для каждого проекта отдельно.

Пример:

```
Текущая задача: реализую авторизацию через JWT.
Проблема: токен не обновляется при истечении срока.
Нужно от ИИ: найти баг в auth/middleware.py и предложить правку.
```

### 7.7 Профили экспорта и AI-пресеты

Профили (существующие, оставить):
- **Быстрый** — краткий обзор, настройка запуска, ключевые файлы.
- **Полный** — все отчёты и AI-контекст.
- **AI-ревью** — оптимизировано для безопасной передачи ИИ.
- **Безопасность** — конфигурация, зависимости, Git, качество кода.
- **Минимальный** — компактный обзор для лёгкой передачи.

AI-пресеты (новые — предустановленные конфигурации под конкретные задачи):
- **Claude Code** — полный контекст, структура, diff, заметки разработчика.
- **ChatGPT** — текстовый дамп в пределах 128K токенов, ключевые файлы.
- **Code Review** — исходники + тесты + структура + Git-история.
- **Security Audit** — конфиги, зависимости, секреты, права доступа.
- **Онбординг разработчика** — README, структура, архитектура, entrypoints.

### 7.8 Дифференциальный экспорт

Режим, при котором в архив попадают только изменения:

- **С момента последнего экспорта** — сравнивает с предыдущим снапшотом.
- **С git-ссылки** — `HEAD~5`, имя ветки, хэш коммита.
- **Только uncommitted** — изменения в рабочем дереве.

Показывает список изменённых файлов до старта. Удалённые файлы фиксируются в манифесте с пометкой `deleted`.

### 7.9 Безопасность

Три режима:

- **Строгий** — исключает `.env`, ключи, токены, бинарники, базы данных, `node_modules`, `venv`, `build`, `dist`. Подходит для передачи ИИ.
- **Сбалансированный** (по умолчанию) — сохраняет код, конфиги, тесты; исключает зависимости и секреты.
- **Полный** — копирует всё; показывает предупреждение; всё равно сканирует секреты.

Сканирование секретов ищет:
- `API_KEY=`, `SECRET=`, `TOKEN=`, `PASSWORD=`, `PRIVATE_KEY`
- `Bearer <токен>`, `BEGIN RSA PRIVATE KEY`
- `AWS_ACCESS_KEY_ID`, `GITHUB_TOKEN`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`
- DATABASE_URL с учётными данными

При обнаружении:
- Строгий / Сбалансированный → файл исключается.
- Полный → явное предупреждение с названием файла (без значения).
- Лог фиксирует факт, но не значение секрета.

### 7.10 Отчёты

Каждый экспорт содержит папку `reports/` с:

| Файл | Содержание |
|---|---|
| `structure_report.txt` | Дерево проекта, количество файлов |
| `git_report.txt` | Ветка, коммиты, незакоммиченные изменения |
| `security_report.txt` | Найденные риски, исключённые файлы |
| `ai_context_report.txt` | AI-контекст: описание, структура, entrypoints |
| `manifest.json` | Список файлов, размеры, checksums, правила |
| `export_summary.txt` | Итоговый читаемый отчёт |
| `text_dump.txt` | Текстовый дамп исходников |
| `export.log` | Читаемый лог |
| `export.jsonl` | Структурированный лог (JSON Lines) |
| `dependency_report.txt` | Анализ зависимостей из package.json / requirements.txt / etc. |
| `metrics_report.txt` | LOC, языки, размеры, сложность |
| `health_score.txt` | Оценка здоровья проекта |
| `dashboard.html` | Интерактивный HTML-дашборд |

### 7.11 Архивация

- ZIP-архив или авто-разбивка на части по 512 МБ.
- Имя включает имя проекта и timestamp: `MyProject_2026-06-28_143022.zip`.
- SHA256-контрольные суммы рядом с архивом.
- Атомарное создание: временный файл → переименование при успехе.
- При отмене или ошибке временные файлы удаляются.

### 7.12 История экспортов

Хранится в `%APPDATA%\ProjectExporterDesktop\history.json`.

Для каждого экспорта: дата, проект, профиль, размер, количество файлов, токены, статус, путь к архиву, предупреждения.

Возможности экрана истории:
- Поиск по проекту и дате.
- Быстрый повторный экспорт с теми же настройками.
- Сравнение двух снапшотов: список добавленных, изменённых, удалённых файлов.
- Открыть папку архива.

### 7.13 Системный трей и интеграция с Windows

- Иконка в системном трее при минимизации окна.
- Контекстное меню трея: «Быстрый экспорт», «Открыть», «Выход».
- Быстрый экспорт последнего проекта без открытия главного окна.
- Watch-режим: следить за папкой проекта и автоматически обновлять дамп при изменении файлов (уведомление в трее).

## 9. Надёжность

Приложение корректно обрабатывает:
- Нет доступа к файлу / файл заблокирован.
- Слишком длинный путь Windows.
- Повреждённая кодировка файла.
- Git отсутствует в системе.
- Недостаточно места на диске.
- Пользователь нажал «Отмена» во время операции.
- Большой проект (10 000+ файлов).
- Символические ссылки и junction-точки Windows.
- Повторный запуск экспорта во время активной операции.

Все ошибки показываются понятным сообщением. Трассировка стека — только в log-файле, не на экране.

## 10. Производительность

- UI не зависает — все долгие операции в worker-потоке.

## 11. Логирование

Логи в трёх местах:
1. В окне приложения в режиме реального времени.
2. В файле `%LOCALAPPDATA%\ProjectExporterDesktop\logs\app.log`.
3. В папке конкретного экспорта (`export.log` + `export.jsonl`).

Уровни: `DEBUG` · `INFO` · `WARNING` · `ERROR` · `CRITICAL`.

Формат JSONL:
```json
{"time":"2026-06-28T14:30:00","level":"info","event":"export_started","project":"my-app"}
{"time":"2026-06-28T14:30:03","level":"warning","event":"secret_file_skipped","path":"<redacted>","rule":"env_file"}
```

Логи не содержат значений секретов. Есть кнопки «Открыть папку логов» и «Скопировать ошибку».

```
  profiles.json       # Пользовательские профили

  cache\              # Кэш (определение стека, токены)
```

- `settings.json` валидируется при запуске; при повреждении создаётся резервная копия.
- История не ломает запуск даже при повреждении.
- Удаление приложения не удаляет данные пользователя без предупреждения.
