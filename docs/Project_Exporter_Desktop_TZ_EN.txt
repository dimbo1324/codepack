TECHNICAL SPECIFICATION
FOR THE DEVELOPMENT OF A WINDOWS DESKTOP APPLICATION
"PROJECT EXPORTER DESKTOP"

Document version: 1.0
Document language: English
Target OS: Windows 10 / Windows 11
Primary stack: Python 3.14 + PySide6 + PyInstaller/Nuitka + Inno Setup
Purpose: to create a safe, structured and AI-ready project snapshot for developers, Claude Code, Codex and other AI-assisted development tools


===============================================================================
1. PROJECT OVERVIEW
===============================================================================

Project Exporter Desktop is a Windows desktop application designed to prepare a software project for transfer to another developer, a team, or an AI-assisted coding tool.

The application must scan a selected project folder, analyse its structure, exclude unsafe and unnecessary files, generate reports, create a text dump of source code where appropriate, and package the result into one archive or a set of archives.

Main product idea:

    "Select a project -> review what will be exported -> safely build an AI-ready archive -> send it to a developer or an AI tool."

The application must be installed through setup.exe, MSI, or an EXE installer and used as a normal Windows program, not as a console script.

The project must be completed to a level where it can be used daily for real work:
- preparing a project for Claude Code;
- preparing a project for Codex;
- handing a project over to another developer;
- creating a safe project snapshot;
- generating reports about structure, Git state, risks, files and AI context.


===============================================================================
2. MAIN GOAL
===============================================================================

Create a high-quality, stable and modern Windows desktop application at a senior+ engineering level that:

1. Installs through setup.exe.
2. Starts as a normal Windows application.
3. Has a professional modern interface.
4. Allows the user to select a project folder.
5. Shows a preview of the export plan.
6. Safely excludes secrets, temporary files, dependency folders and noise.
7. Creates a project archive.
8. Creates reports and AI context.
9. Writes detailed logs.
10. Does not freeze on large projects.
11. Handles errors correctly.
12. Can be compiled and packaged into an installer reproducibly.
13. Has tests, CI and a repeatable release process.


===============================================================================
3. WHAT THE PROJECT MUST NOT DO IN THE FIRST STAGE
===============================================================================

To avoid drifting away from the main implementation path, the first stage must NOT include:

1. Cloud synchronisation.
2. User accounts.
3. An online SaaS service.
4. A server-side backend.
5. Subscriptions or payments.
6. Automatic upload of archives into Claude, Codex or other tools.
7. A custom AI analysis system through external APIs.
8. A code editor.
9. An IDE.
10. A full Git client.
11. A complex plugin system.
12. A team SaaS platform.
13. macOS/Linux support in the first stage.
14. Complex archive encryption in the first stage.
15. Automatic modification of the original source project.

Main focus:

    "A local Windows application that safely and conveniently creates an export snapshot of a project."

6. TARGET TECHNOLOGY STACK
===============================================================================

The target stack:

1. Python 3.14.
2. PySide6 for the graphical user interface.
3. PyInstaller as the primary EXE build tool.
4. Nuitka as an optional/alternative build path after PyInstaller is stable.
5. Inno Setup for Windows setup.exe creation.
6. pytest for unit and integration tests.
7. pytest-qt for GUI tests.
8. ruff for linting and formatting.
9. mypy or pyright for static type checking when the project is ready.
10. GitHub Actions for Windows CI.
11. PowerShell scripts for local build/release workflows.
12. uv or pip-tools for dependency lock files.

Recommended priority:

    PyInstaller first, Nuitka later.

Reason:

    PyInstaller is usually simpler and more predictable for PySide6 desktop packaging. Nuitka may be added as an advanced build path after the application is already stable.

8. ARCHITECTURAL RULES
===============================================================================

1. UI must not contain export business logic.
2. UI must call application services through a clear application context or controller layer.
3. Export must be represented as a pipeline.
4. Every pipeline step must be testable independently.
5. File-system operations must be centralised.
6. Archive operations must be centralised.
7. Exceptions and errors must be typed.
8. Settings must be validated.
9. Constants must be split by responsibility.
10. Secrets must not be stored in source code.
11. GUI, filesystem, Git and archive logic must not be mixed in one file.
12. MainWindow must not become a god object.
13. There must not be one huge constants.py for the whole application.
14. File exclusion rules must not be duplicated in multiple places.
15. Windows path behaviour must be handled explicitly.
16. All long-running work must run outside the UI thread.
17. The export flow must support cancellation.
18. The application must remain usable after non-critical errors.
19. Every major module must have a clear responsibility.
20. The codebase must be understandable by a new developer without personal explanations from the original author.


===============================================================================
9. CORE FUNCTIONAL REQUIREMENTS
===============================================================================

9.1 Project selection

The application must:

1. Select a project root folder.
2. Check that the path exists.
3. Check that the path is a directory.
4. Reject dangerous/system directories, such as:
   - C:\Windows;
   - C:\Program Files;
   - C:\Program Files (x86);
   - C:\Users;
   - drive root directories unless explicitly allowed.
5. Normalise the path.
6. Handle long Windows paths correctly.
7. Store recently selected projects in history.
8. Allow clearing project history.

9.2 Export profiles

The application must support export profiles:

Strict:
- maximum safety;
- excludes secrets, env files, dependency folders, build artifacts, caches and large binaries;
- best for sending to AI tools.

Balanced:
- includes source code and useful context;
- excludes dangerous and heavy files;
- good default mode.

Full:
- includes more project content;
- shows strong warnings;
- still must not include obviously dangerous secrets by default.

Custom:
- user can manually configure rules;
- user can include/exclude paths;
- user can set size limits.

9.3 Export preview

Before export execution, the application must show a preview:

1. Total number of files.
2. Number of included files.
3. Number of excluded files.
4. Total size.
5. Estimated archive size.
6. Large files.
7. Potentially dangerous files.
8. Applied rules.
9. Warnings.
10. Top-level directories.
11. Whether a Git repository exists.
12. Whether there are uncommitted changes.
13. Possible secret indicators.
14. Confirmation button.

The user must be able to review the plan before any archive is created.

9.4 Export execution

The application must create:

1. Temporary staging directory.
2. Safe copy of selected files.
3. Structure report.
4. Git report.
5. Security/safety report.
6. AI context pack.
7. Text dump.
8. Manifest.
9. Checksums.
10. Zip archive or archive set.
11. Export log file.

9.5 Archive output

Archives must:

1. Have clear deterministic names.
2. Include timestamp.
3. Include project name.
4. Support split archive strategy.
5. Have SHA256 checksums.
6. Be created through a temporary file followed by atomic rename.
7. Avoid leaving corrupted archives after errors.
8. Clean temporary data on cancellation.
9. Be portable between Windows computers.

Recommended naming pattern:

    ProjectName_export_YYYYMMDD_HHMMSS.zip
    ProjectName_export_YYYYMMDD_HHMMSS_part01_source.zip
    ProjectName_export_YYYYMMDD_HHMMSS_part02_reports.zip
    ProjectName_export_YYYYMMDD_HHMMSS_part03_ai_context.zip

9.6 Export history

The application must save export history:

1. Export date.
2. Project path.
3. Export profile.
4. Archive size.
5. File count.
6. Status.
7. Archive path.
8. Warnings.
9. Errors, if any.

History must not prevent application startup if the history file is damaged.

11. SECURITY REQUIREMENTS
===============================================================================

Security is a core requirement of this application.

The application must protect the user from accidentally exporting sensitive or dangerous content.

11.1 Files and directories excluded by default

The application must exclude by default:

- .env;
- .env.*;
- .venv;
- venv;
- node_modules;
- dist;
- build;
- .git objects where not needed;
- caches;
- logs;
- temp folders;
- binary build artifacts;
- IDE caches;
- coverage folders;
- test result folders;
- package manager caches;
- private keys;
- certificates;
- database dumps unless explicitly allowed.

11.2 Secret detection

The application must search for suspicious patterns such as:

1. API_KEY=
2. SECRET=
3. TOKEN=
4. PASSWORD=
5. PRIVATE KEY
6. BEGIN RSA PRIVATE KEY
7. Bearer
8. AWS_ACCESS_KEY_ID
9. GITHUB_TOKEN
10. OPENAI_API_KEY
11. ANTHROPIC_API_KEY

The application must not promise perfect secret detection. It must warn that automatic detection is best-effort.

11.3 Path safety

The application must protect against:

1. Symlink traversal.
2. Junction traversal on Windows.
3. Path escape outside project root.
4. Zip Slip risks.
5. Invalid Windows path characters.
6. Long paths.
7. Locked files.
8. Permission errors.
9. Accidental export of system folders.

11.4 User confirmation

If risky files are detected, the user must confirm the export explicitly or adjust rules.


===============================================================================
12. RELIABILITY AND ERROR HANDLING
===============================================================================

The application must expect errors instead of assuming every operation succeeds.

It must handle:

1. No access to a file.
2. File locked by another program.
3. Path too long.
4. Broken encoding.
5. Binary file detected instead of text.
6. Git command error.
7. Git not installed.
8. Archive creation error.
9. Not enough disk space.
10. User cancellation.
11. Staging directory cleanup failure.
12. Broken settings.json.
13. Broken history.json.
14. Very large project.
15. Symbolic links.
16. Windows junctions.
17. Repeated export launch while another export is active.

Error rules:

1. User-facing messages must be clear and short.
2. Developer logs must contain diagnostic details.
3. Tracebacks must not be the main user interface.
4. Non-critical report failures should not always fail the full export.
5. Critical archive corruption must fail the export.
6. Cancellation must be treated as a controlled state, not a crash.
7. Cleanup should be attempted after failure.
8. The application must remain usable after errors.

15. PERFORMANCE REQUIREMENTS
===============================================================================

The application must work with:

1. Small projects.
2. Large projects.
3. Large text files.
4. Partially inaccessible files.
5. Long archive creation.

Performance rules:

1. UI must not freeze.
2. Long operations must run in a worker thread.
3. Progress must update regularly.
4. Cancellation must be checked in long loops.
5. Large files must be read in streaming mode.
6. Text dump must have limits.
7. Binary files must not be read as text.
8. Repeated calculations should be minimised.
9. The whole project must not be kept in RAM unnecessarily.
10. Archive creation must be streamed where possible.

Recommended limits:

- maximum bytes per text file;
- maximum total text dump size;
- maximum archive part size;
- maximum number of warnings shown directly in UI;
- detailed report for the full list.

28. CONCLUSION
===============================================================================

Project Exporter Desktop must become a professional Windows application for creating safe, structured and AI-ready project snapshots.

The key value of the project is not simply making a zip file. The key value is creating a controlled, explainable and safe export process:

    project scan -> export preview -> safety review -> reports -> text dump -> archive -> logs -> handoff to developer or AI tool.

The project must remain focused. It should not become an IDE, SaaS service, cloud synchronisation tool or AI platform in the first stage.

The best implementation path is:

1. Stabilise baseline.
2. Decompose architecture.
3. Harden export pipeline.
4. Improve safety and preview.
5. Improve UI/UX.
6. Build reproducible EXE.
7. Build Inno Setup installer.
8. Add packaging and release tests.
9. Prepare stable Windows release.

Final quality criterion:

    The application can be safely changed, maintained, installed, used daily and handed over to another developer without fear.
