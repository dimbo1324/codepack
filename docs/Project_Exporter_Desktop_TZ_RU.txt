НА РАЗРАБОТКУ WINDOWS DESKTOP-ПРИЛОЖЕНИЯ
"PROJECT EXPORTER DESKTOP"

Версия документа: 1.0
Язык документа: русский
Целевая ОС: Windows 10 / Windows 11
Основной стек: Python 3.14 + PySide6 + PyInstaller/Nuitka + Inno Setup
Назначение: создание безопасного, структурированного и AI-ready снимка проекта для разработчиков, Claude Code, Codex и других ИИ-инструментов


===============================================================================
1. КРАТКОЕ ОПИСАНИЕ ПРОЕКТА
===============================================================================

Project Exporter Desktop — это настольное приложение для Windows, предназначенное для подготовки разработческого проекта к передаче человеку, команде или ИИ-инструментам.

Приложение должно сканировать выбранную папку проекта, анализировать её структуру, исключать опасные и ненужные файлы, формировать отчёты, создавать текстовый дамп кода и собирать результат в архив или набор архивов.

Главная идея приложения:

    "Выбрать проект -> проверить, что попадёт в экспорт -> безопасно собрать AI-ready архив -> передать разработчику или нейросети".

Приложение должно быть установлено через setup.exe или MSI/EXE-инсталлятор и использоваться как полноценная Windows-программа, а не как скрипт из консоли.

Проект должен быть доведён до состояния, в котором им можно пользоваться ежедневно для реальных задач:
- подготовка проекта для Claude Code;
- подготовка проекта для Codex;
- передача проекта другому разработчику;
- создание безопасного снимка проекта;
- создание отчётов о структуре, Git-состоянии, рисках, файлах и контексте.

===============================================================================

2. UI вызывает сервисы через понятный ApplicationContext.

5. Работа с файлами должна быть централизована.
6. Работа с архивами должна быть централизована.

8. Настройки должны валидироваться.
9. Константы должны быть разделены по смыслу.
10. Нельзя хранить секреты в коде.
11. Нельзя смешивать GUI, filesystem, Git и archive logic в одном файле.
12. Нельзя делать “god object” MainWindow.
13. Нельзя делать огромный constants.py на всё приложение.
14. Нельзя дублировать правила исключения файлов в разных местах.
15. Нельзя полагаться на случайное поведение путей Windows.


===============================================================================
9. ОСНОВНЫЕ ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
===============================================================================

9.1 Выбор проекта

Приложение должно позволять:

1. Выбрать корневую папку проекта.
2. Проверить, что путь существует.
3. Проверить, что путь является директорией.
4. Запретить выбор системных директорий:
   - C:\Windows;
   - C:\Program Files;
   - C:\Program Files (x86);
   - C:\Users;
   - корень диска C:\ без явного подтверждения.
5. Нормализовать путь.
6. Корректно обрабатывать длинные пути Windows.
7. Сохранять последние выбранные проекты в историю.

9.2 Профили экспорта

Должны быть профили:

Strict:
- максимально безопасный профиль;
- исключает env, secrets, node_modules, venv, dist, build, binaries;
- подходит для передачи ИИ.

Balanced:
- основной режим по умолчанию;
- сохраняет исходный код, конфиги, тесты, документацию;
- исключает зависимости, build output, secrets.

Full:
- максимально полный экспорт;
- требует предупреждения;
- всё равно исключает критичные секреты по safety policy.

Custom:
- пользователь может включать/выключать категории файлов;
- пользователь может добавлять собственные exclude rules.

9.3 Предварительный план экспорта

Перед созданием архива приложение должно показать:

1. Количество всех файлов.
2. Количество включённых файлов.
3. Количество исключённых файлов.
4. Общий размер.
5. Ожидаемый размер архива.
6. Крупные файлы.
7. Потенциально опасные файлы.
8. Применённые правила.
9. Предупреждения.
10. Список top-level директорий.
11. Наличие Git-репозитория.
12. Наличие uncommitted changes.
13. Наличие подозрительных секретов.
14. Кнопку подтверждения.

9.4 Экспорт

Приложение должно создавать:

1. Рабочую staging-директорию.
2. Копию выбранных файлов.
3. Структурный отчёт.
4. Git-отчёт.
5. Security/safety отчёт.
6. AI context pack.
7. Text dump.
8. Manifest.
9. Checksums.
10. Zip-архив или набор архивов.
11. Log-файл экспорта.

9.5 Архивация

Архивы должны:

1. Иметь понятные имена.
2. Содержать timestamp.
3. Содержать имя проекта.
4. Поддерживать split archive strategy.
5. Иметь SHA256 checksums.
6. Создаваться через временный файл с atomic rename.
7. Не оставлять повреждённые архивы при ошибке.
8. Очищать временные данные при отмене.
9. Быть переносимыми между Windows-компьютерами.

9.6 История экспортов

Приложение должно хранить локальную историю:

1. Дата экспорта.
2. Проект.
3. Профиль.
4. Размер архива.
5. Количество файлов.
6. Статус.
7. Путь к архиву.
8. Предупреждения.
9. Ошибки, если были.

История должна храниться в AppData пользователя.

10. ОТЧЁТЫ
===============================================================================

Каждый экспорт должен содержать папку reports.

Минимальный набор отчётов:

1. structure_report.txt
   - дерево проекта;
   - количество файлов;
   - основные директории.

2. git_report.txt
   - branch;
   - latest commits;
   - uncommitted changes;
   - ignored files summary;
   - remote info без секретов.

3. security_report.txt
   - найденные потенциальные секреты;
   - исключённые опасные файлы;
   - подозрительные расширения;
   - предупреждения.

4. ai_context_report.txt
   - краткое описание проекта;
   - структура;
   - важные файлы;
   - entry points;
   - как читать архив.

5. manifest.json
   - список включённых файлов;
   - размеры;
   - checksums;
   - applied rules.

6. export_summary.txt
   - итоговый человекочитаемый отчёт.

7. text_dump.txt
   - безопасный текстовый дамп исходников и важных файлов.

8. logs/export.log
   - человекочитаемый лог.

9. logs/export.jsonl
   - структурированный machine-readable лог.

11. БЕЗОПАСНОСТЬ
===============================================================================

Приложение должно защищать пользователя от случайной передачи секретов.

Обязательно исключать или предупреждать:

1. .env
2. .env.local
3. .env.production
4. *.pem
5. *.key
6. *.pfx
7. *.p12
8. id_rsa
9. id_ed25519
10. credentials.json
11. secrets.*
12. token.*
13. private.*
14. node_modules
15. .venv
16. venv
17. __pycache__
18. dist
19. build
20. .git/objects
21. large binaries
22. database dumps
23. *.sqlite
24. *.db
25. browser profiles
26. cache directories.

Safety scan должен искать паттерны:

1. API_KEY=
2. SECRET=
3. TOKEN=
4. PASSWORD=
5. PRIVATE KEY
6. BEGIN RSA PRIVATE KEY
7. Bearer
8. AWS_ACCESS_KEY_ID
9. GITHUB_TOKEN
10. OPENAI_API_KEY
11. ANTHROPIC_API_KEY

Если обнаружен риск:

- Strict/Balanced режим должен исключить файл;
- Full режим должен показать явное предупреждение;
- лог должен фиксировать факт исключения, но не значение секрета.


===============================================================================
12. НАДЁЖНОСТЬ И ОБРАБОТКА ОШИБОК
===============================================================================

Приложение должно корректно обрабатывать:

1. Нет доступа к файлу.
2. Файл заблокирован другой программой.
3. Слишком длинный путь.
4. Повреждённая кодировка.
5. Бинарный файл вместо текстового.
6. Ошибка Git.
7. Git отсутствует в системе.
8. Ошибка архивации.
9. Недостаточно места на диске.
10. Пользователь отменил операцию.
11. Staging directory не очищена.
12. Ошибка чтения settings.json.
13. Ошибка записи history.json.
14. Большой проект.
15. Символические ссылки.
16. Junctions на Windows.
17. Повторный запуск экспорта во время активной операции.

Пользователь должен видеть понятное сообщение.

Разработчик должен иметь подробный log.


===============================================================================
13. ЛОГИРОВАНИЕ
===============================================================================

Логи должны быть:

1. В интерфейсе в режиме реального времени.
2. В файле AppData.
3. В папке конкретного экспорта.
4. Разделены по уровням:
   - DEBUG;
   - INFO;
   - WARNING;
   - ERROR;
   - CRITICAL.
5. Не содержать секреты.
6. Поддерживать копирование ошибки.
7. Поддерживать кнопку "Открыть папку логов".

Форматы:

1. export.log — обычный текст.
2. export.jsonl — структурированный JSON Lines.

Пример JSONL:

{"time":"2026-06-09T12:00:00","level":"info","event":"export_started","project":"demo"}
{"time":"2026-06-09T12:00:03","level":"warning","event":"secret_file_skipped","path":"<redacted>","rule":"env_file"}

===============================================================================
15. ПРОИЗВОДИТЕЛЬНОСТЬ
===============================================================================

Приложение должно выдерживать:

1. 10 000+ файлов.
2. Большие проекты.
3. Большие текстовые файлы.
4. Частично недоступные файлы.
5. Долгую архивацию.

Требования:

1. UI не должен зависать.
2. Долгие операции выполняются в worker thread.
3. Progress обновляется регулярно.
4. Отмена проверяется в длинных циклах.
5. Большие файлы читаются потоково.
6. Text dump имеет лимиты.
7. Бинарные файлы не читаются как текст.
8. Повторные вычисления минимизированы.
9. Нет утечек памяти из-за хранения всего проекта в RAM.
